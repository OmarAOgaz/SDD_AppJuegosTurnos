## Exploration: Lobby Player Controls Redesign

### Current State

The lobby (`lib/features/lobby/lobby_screen.dart`) is a single `ConsumerStatefulWidget` that renders two very different layouts depending on `widget.role`:

- **Host** (`_buildHost`): a `ListView` of read-only `ListTile` player rows (`_playerTile`) — `CircleAvatar` colored by `colorId`, `displayName`, a subtitle with `slotNumber` + connection status, and a trailing "Tú" tag for self. Below the list is room config (name, turn duration, round increment, variable order toggle, max players) editable only by the host. There is **no reorder UI at all today** — `LobbyRules.tryReorderSlots` / `tryReorderTurnSequence` and `HostRoomController.reorderSlots` / `reorderTurnSequence` exist and are fully wired end-to-end (host-local call → mutate `GameRoom.slots`/`turnSequence` → `_broadcastLobbyState()`), but nothing in the UI calls them yet.
- **Client** (`_buildClient`): the same read-only `_playerTile` list for all seated players, plus a separate "Tu perfil" section (`_clientSelfEditor`) with three `DropdownButtonFormField`s (name, color, sound) that only edit the **local** player, sending `UPDATE_PLAYER` over the socket.

Color/sound assignment logic already implements the "UI-only exclusivity" decision recorded in Engram (`sdd/mvp-lobby-turn-timer/lobby-exclusivity`, obs #66): taken ids are **filtered out** of the dropdown via `eligibleColorIds` / `eligibleSoundIds` (`lib/core/domain/eligible_picker.dart`), which return free ids **plus the player's own current id**. `LobbyRules.tryUpdatePlayer` silently ignores a color/sound change if the target id is taken by someone else (no rejection message to the player) — this matches "no `UPDATE_PLAYER_REJECTED` for color/sound" from the locked spec (`openspec/specs/lobby/spec.md`, Requirement "UPDATE_PLAYER with UI-only color/sound exclusivity"). Duplicate `displayName`s across players are explicitly allowed by that same requirement.

Reordering is **already architecturally host-only at the protocol level**, not just a UI convention: `lib/core/network/game_socket_client.dart` (the client-side socket wrapper) exposes only `sendJoin`, `sendLeave`, `sendUpdatePlayer` — there is no `sendReorderSlots`/`sendReorderTurnSequence` method, and the host applies reorders as a direct in-process call to its own `GameRoom` (no wire message needed for the host's own mutation). A malicious/modified client literally has no message type to request a reorder. This means the requested "only the host can see and use the reordering" constraint requires no protocol change — it is a pure UI/host-role gating concern, already covered by `widget.role == 'host'` (`_isHost`).

Catalogs are small and fixed: `ColorCatalog` has exactly 8 colors, `SoundCatalog` has exactly 8 sounds (all currently mapped to the same silent stub asset `assets/sounds/sound_stub.wav` — no distinguishable audio yet), and `RoomConfig.maxPlayersLimit == 8`, so the catalogs exactly cover the max seat count with no spare/overflow option.

There is no widget-test coverage for `lobby_screen.dart` or `personalize_screen.dart` today (`test/features/` only has `ended_screen_smoke_test.dart` and `game_screen_feedback_test.dart`); all current lobby test coverage is pure-logic (`test/core/domain/lobby_rules_test.dart`, `eligible_picker_test.dart`, `preference_assignment_test.dart`). No `flutter analyze`/`flutter test` runner has been confirmed available in this sandbox during this exploration (not invoked, per explore-phase "no implementation" rule), so this is asserted from `openspec/testing-capabilities.md` planning notes, not verified live.

Flutter's SDK already ships `ReorderableListView`/`ReorderableDragStartListener`, so drag-and-drop reordering needs **no new pubspec dependency** — `pubspec.yaml` has no `intl`/`flutter_localizations` either (all UI strings are hardcoded Spanish), so localization is out of scope for this redesign regardless.

### Affected Areas

- `lib/features/lobby/lobby_screen.dart` — primary target. `_buildHost`, `_buildClient`, `_playerTile`, `_clientSelfEditor` are all replaced/rewritten by a new per-player row: `Jugador N` label above a rounded-rect colored box, trailing `Color` button + circular sound icon, leading reorder arrows (host-only) + drag handle.
- `lib/core/domain/eligible_picker.dart` — today's `eligibleColorIds`/`eligibleSoundIds` **filter out** taken ids. The new palette needs to show **all 8** ids with a "taken" flag (struck-through, non-tappable) instead of hiding them. Likely needs a new helper (e.g. returning `{id, isTaken}` pairs) alongside — not necessarily replacing — the existing eligible-ids functions (still useful elsewhere, e.g. `JOIN` preference assignment in `preference_assignment.dart` is unaffected).
- `lib/core/domain/lobby_rules.dart` — `takenColorIds`/`takenSoundIds` (already `Set<String>` derived from `room.seatedPlayers()`) are the exact input the new palette needs; `tryUpdatePlayer`'s silent-ignore-on-taken behavior stays unchanged (client should now visually prevent the tap in the first place, consistent with the existing UI-only exclusivity decision).
- `lib/server/host_room_controller.dart` — `reorderSlots()` / `reorderTurnSequence()` already exist and broadcast `LOBBY_STATE`; the new UI is the first caller. Needs a product decision on whether the new arrows/drag move `slots` (seating/display order) only, `turnSequence` (play order) only, or both together (see Ambiguity #1).
- `lib/core/catalogs/color_catalog.dart`, `lib/core/catalogs/sound_catalog.dart` — read-only catalogs, no change needed; `CatalogColor.color` / `CatalogSound.displayName` feed the new palette swatches/list directly.
- `lib/core/models/player.dart`, `lib/core/models/game_room.dart` — no model changes anticipated; `slotNumber` already exists and is exactly the "N" in "Jugador N" once slot-order semantics are confirmed.
- `lib/features/player_profile/personalize_screen.dart` — not in scope per the request, but it currently uses plain `DropdownButtonFormField` pickers for the same two catalogs. If the new lobby palette becomes a shared widget, this screen is a natural (optional) follow-up for visual consistency — flagging only, not proposing to touch it now.
- `test/features/lobby/` (new) — no existing widget tests to extend; this redesign is effectively greenfield for lobby UI testing. `test/core/domain/lobby_rules_test.dart` and `eligible_picker_test.dart` remain valid for any new pure-logic helper (e.g., "all ids with taken flag").
- `openspec/specs/lobby/spec.md` — the "UPDATE_PLAYER with UI-only color/sound exclusivity" requirement's scenario "Taken colors omitted from picker" (`color_2 MUST NOT appear as an eligible choice`) needs a **delta**: the new behavior is "taken colors/sounds MUST remain visible but MUST render struck-through/disabled," which is a change in wording, not in the underlying exclusivity guarantee. This should be handled as a `MODIFIED` requirement in `sdd-spec`, not silently drifted.

### Approaches

1. **In-place rewrite of `lobby_screen.dart` with new stateless row/palette widgets, reusing existing domain/network layer as-is** — build `_PlayerRow`, `_ColorPaletteButton`, `_SoundPaletteButton`, `_ReorderControls` as private widgets inside (or split out from) `lobby_screen.dart`; add one small domain helper for "catalog id + taken flag" pairs; wire arrows/drag to the already-existing `HostRoomController.reorderSlots`/`reorderTurnSequence`.
   - Pros: No protocol/model changes; smallest blast radius; matches existing architecture (host-authoritative, `LOBBY_STATE` still the single source of truth); reuses `ColorCatalog`/`SoundCatalog`/`LobbyRules.takenColorIds`/`takenSoundIds` verbatim; testable with plain widget tests (no `flutter_test` new deps).
   - Cons: `lobby_screen.dart` (already ~475 lines) grows further unless private widgets are extracted to their own files; still need to resolve reorder-target ambiguity (slots vs turnSequence) before implementation; drag gesture vs. tap-target conflicts (arrows/Color/Sound buttons living in the same draggable row) need careful `ReorderableDragStartListener` scoping to a dedicated handle, not the whole row.
   - Effort: Medium.

2. **Extract a shared `PlayerSeatCard` + `PreferencePaletteSheet` widget pair into `lib/features/lobby/widgets/`, used by both host and client builds, and later reusable by `personalize_screen.dart`** — same underlying logic as Approach 1, but designed for reuse from day one.
   - Pros: Cleaner file sizes; sets up future consistency between lobby and Personalización pickers without extra cost now; easier to unit-test each widget in isolation.
   - Cons: Slightly more upfront design/naming work; touches more new files for a feature that, per the request, is scoped to the lobby only; risk of over-engineering for a single call site today.
   - Effort: Medium-High.

3. **Minimal visual patch only (keep dropdowns, just add rounded-rect color swatch + reorder icons without a real palette popover)** — satisfies the letter of "show color as background" and "arrows to reorder" but keeps `DropdownButtonFormField` for color/sound instead of a custom palette with strike-through taken items.
   - Pros: Fastest; lowest risk; no new gesture-handling code.
   - Cons: Does not satisfy the explicit requirement that taken colors/sounds remain **visible but crossed-out** inside a dedicated `Color`/sound-icon control — dropdowns in Flutter don't natively support a "visible-but-disabled-with-strikethrough" item render without custom `DropdownMenuItem` builders, which ends up being nearly as much work as a real palette while looking worse. Not recommended.
   - Effort: Low, but rejected on requirements-fit grounds.

### Recommendation

**Approach 1** (in-place rewrite, small new domain helper, no protocol changes), with private widgets split into a `lib/features/lobby/widgets/` sub-folder purely for file-size hygiene (not full Approach 2's reuse-first scope) — reuse across `personalize_screen.dart` can be a follow-up change if the user wants it later. This keeps the change tightly scoped to the lobby, matches the existing host-authoritative architecture with zero wire-protocol changes, and lets `sdd-spec` express the palette-visibility change as a focused `MODIFIED` delta on the existing `lobby` spec domain.

### Risks

- **Ambiguity on reorder target** (`slots` vs `turnSequence` vs both) must be resolved before `sdd-propose`/`sdd-design` — implementing the wrong one silently breaks the "Requirement: Host reorder slots and turn sequence" contract or produces a UI that reorders seats without reordering turn play (or vice versa), confusing under `variableTurnOrder = false` where slot order and turn order are expected to track together.
- **No existing widget tests for `lobby_screen.dart`** — this redesign is high risk to regress silently (e.g., breaking the client's "Tu perfil" self-edit flow) without new coverage; `sdd-tasks`/`sdd-apply` should budget for adding `test/features/lobby/` widget tests, especially for host-only gating of reorder controls and taken-color/sound strike-through rendering.
- **Drag-and-drop + inline action buttons in the same row** is a known Flutter UX pitfall — `ReorderableListView` captures the whole tile's vertical drag region by default; without a scoped `ReorderableDragStartListener` on a dedicated handle icon, dragging can accidentally trigger (or block) taps on the adjacent Color/Sound buttons or arrows.
- **Sound catalog has no distinguishable audio yet** (`SoundCatalog` — all 8 ids map to the same stub `.wav`) — if the new sound picker is expected to let players preview-tap a sound, there is currently nothing audibly different to preview; this may be fine (visual/label-only picker) or may surface as a confusing "silent" preview during manual QA. Needs a product answer, not a code blocker.
- **Spec drift risk**: the `openspec/specs/lobby/spec.md` scenario "Taken colors omitted from picker" literally says taken colors "MUST NOT appear as an eligible choice" — the new requirement (visible-but-crossed-out) must be captured as an explicit `MODIFIED` delta in `sdd-spec`, or the archived spec will contradict the shipped UI.
- **400-line PR budget**: a full row redesign (new palette popovers/sheets for both color and sound, reorder arrows, drag-and-drop, host-only gating, plus new/updated widget tests) will very likely exceed the 400-line review budget in a single PR. `sdd-tasks` should plan a chained-PR split (e.g., PR1: player row visual redesign (rounded box + "Jugador N" label, no interactivity change) → PR2: color/sound palette with taken/strike-through → PR3: host-only reorder arrows + drag-and-drop). Flagging now per the review-workload guard so `sdd-propose`/`sdd-tasks` cache the decision early (`delivery_strategy` is already cached as `ask-on-risk`).

### Open Product Questions (recommend resolving in `sdd-propose`, before `sdd-design`)

1. Do the reorder controls move **`slots`** (seat/display order), **`turnSequence`** (play order), or **both together**? (Both exist independently in `LobbyRules`/`HostRoomController`.)
2. Does **"Jugador N"** always reflect current `slotNumber` (so it changes live when the host reorders), and where does each player's **custom `displayName`** render relative to that fixed label — inside the colored box, below it, or is the free-text name edit moved elsewhere entirely?
3. Are the inline **Color**/sound controls interactive **only on the local player's own row** (host sees everyone's box but can only recolor their own seat, same as today's "Tu perfil" self-only edit), or does the host get to override *any* seat's color/sound from their device?
4. What **widget shape** should the "Color"-button palette and the sound-icon palette open as — modal bottom sheet, popover/`Overlay`, or full dialog? (Materially changes implementation effort and is worth locking before `sdd-design`.)
5. Is a **sound preview** (tap-to-hear) expected in the new sound picker, given all 8 catalog sounds currently share one silent stub asset?
6. Should the existing read-only connection status ("conectado"/"desconectado") indicator be preserved somewhere in the new row layout, or is it intentionally dropped from this redesign?

### Ready for Proposal

**Yes, with caveats** — the codebase fully supports this change with no protocol/model additions (reorder plumbing and taken-id tracking already exist), so `sdd-propose` can proceed. The orchestrator should surface the 6 open product questions above to the user before or during `sdd-propose`, since at least Q1 (reorder target) and Q2 (label vs. name placement) materially change the shape of the delta spec and design, and Q4 affects effort estimation for `sdd-tasks`. Recommend locking these as explicit assumptions in the proposal rather than deferring to `sdd-design`.
